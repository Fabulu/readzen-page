# Build scripts

Small Node.js utilities that generate static assets checked into the repo.
They run offline and have no runtime dependencies beyond a modern Node.js
(18+, ES modules).

## `build-dict-shards.js`

Slices the CC-CEDICT dictionary into per-character JSON shards so the
dictionary view can fetch only the tiny bucket it needs instead of the full
3 MB corpus.

### Input

By default: `C:/programmieren/MergeWorkCbeta/CBETA-Translator/Assets/Dict/cedict_ts.u8`

Override with the `CEDICT_PATH` env var:

```bash
CEDICT_PATH=/path/to/cedict_ts.u8 node build/build-dict-shards.js
```

### Output

- `dict/{char}.json` — one file per unique first character of the traditional
  form. Each file is a JSON array of `{ trad, simp, pinyin, defs }` entries.
- `dict/_index.json` — manifest listing every shard character and a build
  timestamp.

Existing `dict/*.json` files are cleared before each run so deleted entries
don't stick around.

### When to re-run

Only when CC-CEDICT itself is updated. The CEDICT source is checked into the
CBETA-Translator desktop app's `Assets/Dict/` directory; when that file
changes upstream, re-run this script to regenerate the shards and commit
the result.

```bash
node build/build-dict-shards.js
git add dict/
git commit -m "Refresh CC-CEDICT shards"
```

## `build-pagefind-index.js`

Builds a [Pagefind](https://pagefind.app) full-text search index from the CBETA
and OpenZen XML corpus. Enables client-side full-text search across ~5000
Buddhist texts with CJK support.

### Prerequisites

```bash
npm install   # installs pagefind package
```

Corpus repos must be on disk (sibling directories):
- `CbetaZenTexts/xml-p5/` — CBETA source XML (~5000 texts)
- `OpenZenTexts/xml-open/` — OpenZen source XML
- `CbetaZenTranslations/titles.jsonl` — title metadata
- `CbetaZenTranslations/xml-p5t/` — translated files (filter metadata)
- `CbetaZenTranslations/zen_texts.json` — zen text IDs
- `OpenZenTranslations/titles.jsonl` — OpenZen title metadata

### How to run

```bash
npm run build:search
# or: node build/build-pagefind-index.js
```

### Output

`pagefind/` directory (~30MB, ~7500 files). This is **gitignored** — rebuild
locally or during deployment. Contents:

- `pagefind.js` — client entry point
- `wasm.*.pagefind` — WASM search engine (includes CJK segmenter)
- `index/` — sharded inverted index (~2500 files, loaded on demand)
- `fragment/` — per-document excerpts (~5000 files)
- `filter/` — faceted filter data

### What it indexes

1. All CBETA XML source texts (Chinese, `language: "zh"`)
2. All OpenZen XML source texts (Chinese)
3. Filters per text: `corpus`, `translated`, `zen`, `collection`

### How it works at runtime

The SPA lazy-loads Pagefind when the user picks "Full Text" search mode.
Only the index shards needed for each query are downloaded (~200KB–2MB
per search). The WASM engine runs entirely in the browser.

### Timing

- ~10–15 minutes for ~5000 files
- ~4GB RAM peak during index construction
- One-time build; output is static

### When to rebuild

- New texts added to corpus
- Translations updated (affects `translated` filter)
- Pagefind version bumped

### Configuration

Paths are at the top of `build-pagefind-index.js`. Adjust if layout differs.

## `shard-master-corpus.js`

Splits `master-corpus.json` (monolithic, ~750KB) into per-master JSON files
so the SPA fetches only the relevant master's data (~3KB median).

### How to run

```bash
node build/shard-master-corpus.js
```

### Input

`CbetaZenTranslations/master-corpus.json` — generated by the ReadZen
desktop app's corpus scan (276+ masters across ~5000 texts).

### Output

Written to `CbetaZenTranslations/corpus/masters/`:
- `_index.json` — lightweight manifest (~15KB) with counts per master
- `{slug}.json` — one file per master with all text appearances

### When to rebuild

After the desktop app regenerates `master-corpus.json` (e.g., when
Zen master data changes or new texts are added to the corpus).

## `bundle-dict-buckets.js`

Bundles the per-character dict shards (`dict/{char}.json`, ~12K files) into
numbered bucket files (`dict/{N}.json`, 201 files) plus a manifest
(`dict/_manifest.json`). This keeps the Cloudflare Pages deployment under
the 20K file limit.

### How to run

```bash
node build/bundle-dict-buckets.js
```

### Input

Existing per-character shards in `dict/` (generated by `build-dict-shards.js`).

### Output

- `dict/_manifest.json` — maps each character to its bucket number (codePoint % 201)
- `dict/{0-200}.json` — each bucket is a JSON object keyed by character, containing entry arrays

### When to rebuild

After running `build-dict-shards.js` (i.e., when CC-CEDICT is updated). Run
the shard script first, then the bucket script:

```bash
node build/build-dict-shards.js
node build/bundle-dict-buckets.js
git add dict/
git commit -m "Refresh CC-CEDICT buckets"
```
